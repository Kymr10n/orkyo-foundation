# 04 — Frontend and API Impact

## Objective

Update frontend and API integration so floorplans are loaded through authenticated asset endpoints instead of static filesystem paths.

## API contract

Recommended metadata DTO:

```json
{
  "id": "uuid",
  "ownerType": "location_node",
  "ownerId": "uuid",
  "assetType": "floorplan",
  "fileName": "floorplan.png",
  "contentType": "image/png",
  "sizeBytes": 123456,
  "checksumSha256": "...",
  "createdAt": "2026-05-21T00:00:00Z",
  "updatedAt": "2026-05-21T00:00:00Z",
  "downloadUrl": "/api/assets/{assetId}"
}
```

The `downloadUrl` may be generated by the frontend if that is the current pattern. Prefer backend-provided URLs only if that aligns with existing API style.

## Frontend changes

Find all places where the UI assumes a static path or local filesystem-backed URL.

Replace with:

```text
asset metadata -> authenticated API download URL -> image/pdf rendering
```

## Floorplan rendering

For image floorplans:

```tsx
<img src={assetDownloadUrl} alt="Floorplan" />
```

If authorization headers are required and cookies are not used, `<img>` may not work directly. In that case:

1. fetch asset as Blob using authenticated API client
2. create object URL with `URL.createObjectURL(blob)`
3. use object URL as image source
4. revoke object URL on unmount

Choose based on Orkyo's current auth/session model.

## Upload flow

Use `multipart/form-data`.

Recommended UX behavior:

- validate file type client-side for early feedback
- validate file size client-side for early feedback
- server remains source of truth
- show progress if upload component already supports it
- refresh owner asset metadata after upload

## Delete/replace flow

For floorplans, replacement should be simple:

- user uploads new floorplan
- backend replaces existing floorplan asset for that owner
- frontend invalidates relevant query/cache
- new image is displayed

If React Query is used, invalidate:

```text
asset metadata query for owner
location/site/space detail query if it embeds asset metadata
```

## API client

Add or update asset client functions:

```ts
uploadAsset(ownerType, ownerId, assetType, file)
listAssetsForOwner(ownerType, ownerId)
downloadAsset(assetId)
deleteAsset(assetId)
```

If Orkyo has generated API clients or central fetch wrappers, follow those conventions.

## Cache behavior

When asset checksum changes, the rendered image should refresh.

Options:

- append checksum as query parameter: `/api/assets/{id}?v={checksum}`
- use ETag and normal browser cache behavior
- React Query invalidation after upload/delete

Recommended first approach:

```text
use ETag + query invalidation
```

Use checksum query parameter only if the browser keeps stale images.

## UI copy

Keep it simple:

- Upload floorplan
- Replace floorplan
- Remove floorplan
- Maximum file size: configured limit
- Supported file types: PNG, JPEG

## Error handling

Map backend validation errors to user-friendly messages:

- file too large
- unsupported file type
- upload failed
- floorplan not found
- not authorized

## Compatibility phase

During migration bridge phase, existing floorplans may still be served through legacy URLs.

Frontend should ideally consume a unified field from the backend:

```json
{
  "floorplan": {
    "assetId": "uuid-or-null",
    "downloadUrl": "string-or-null",
    "isLegacy": false
  }
}
```

This keeps migration logic out of the frontend where possible.

## Documentation updates

Update user/admin/deployment documentation:

- floorplans are stored in PostgreSQL
- backup PostgreSQL to back up floorplans
- asset folder mount is no longer required
- size/type limits are configurable
